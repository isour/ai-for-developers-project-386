# Правила репозитория

Инструкции для участников и агентов с кодом в этом монорепозитории. Установка, демо и длинные гайды — в README; здесь — кратко и по делу.

## Продукт: календарь звонков

Веб-приложение **без регистрации и авторизации** — только планирование звонков (встреч) через общий календарь.

**Роли**

- **Владелец календаря** — задаёт доступное время для встреч, управляет **типами событий**, видит **все предстоящие бронирования** в одном списке.
- **Гость** — выбирает тип события и бронирует **свободный слот** в рамках окна записи.

**Возможности владельца**

- Создавать **типы событий** с постоянным **идентификатором**, **названием**, **описанием** и **длительностью в минутах**.
- Просматривать страницу **предстоящих встреч** со списком бронирований по **всем** типам событий.

**Возможности гостя**

- Открыть список типов событий (название, описание, длительность).
- Выбрать тип, открыть календарь, выбрать **свободный слот** на **ближайшие 14 дней** от текущей даты и оформить бронирование.

**Бизнес-правила**

- **Одна запись на одно время:** не больше одного бронирования на одно и то же время; правило действует для **всех** типов событий (единая модель занятости календаря).
- **Окно записи по умолчанию:** слоты строятся на **14 дней**, начиная с **сегодня**. Гость может записаться только внутри этого окна.

**Вне продуктового описания**

- Учётные записи, логин, пароли, OAuth и RBAC **внутри приложения** не используются. Если позже потребуется отличать владельца от гостя без аккаунтов (секретные ссылки, токены, отдельные URL) — **одной строкой** опишите выбранный механизм здесь.

## Структура репозитория

| Область | Путь | Примечание |
|--------|------|------------|
| API на Go | `backend/` | HTTP, хранение, правила слотов и бронирований |
| Веб-интерфейс | `frontend/` | React, TypeScript, Vite; **Feature-Sliced Design (FSD)** |
| Контракты | `contracts/` или `typespec/` (имя зафиксировать при скеффолдинге) | Исходники **TypeSpec**; собранный **OpenAPI** для Prism и при необходимости генераторов кода |
| Образы / compose | корень репозитория (`Dockerfile*`, `docker-compose.yml` и аналоги) | Полный стек локально и близко к продакшену |
| Автоматизация | **`Makefile`** в корне | Единая точка входа: `fmt`, `lint`, `build`, `test`, `pre-commit` |

Не дублируйте длинные инструкции по установке; при необходимости отсылайте к README.

## Бэкенд (Go)

Ориентир — принятая в сообществе раскладка, близкая к **[Standard Go Project Layout](https://github.com/golang-standards/project-layout)**:

- `backend/cmd/<сервис>/main.go` — точки входа (бинарники).
- `backend/internal/...` — HTTP-handlers, доменная логика, адаптеры хранения. Внешние модули **не** должны импортировать чужие деревья `internal` без необходимости.

По возможности направляйте зависимости так: handlers → сценарии использования / application → repository; без навязывания тяжёлого фреймворка сверх потребностей кода.

**Форматирование:** `gofmt` / `go fmt` и **`goimports`**. Для файлов **`.go`** на **Prettier** не опираться — в экосистеме Go это не норма.

**Линтинг:** **`golangci-lint`** (govet, staticcheck, errcheck и др.). Конфиг — в `backend/` или в корне, согласованно с CI.

## Фронтенд (React + TypeScript + Vite)

- **UI:** **shadcn/ui** поверх React.
- **Архитектура:** **FSD** — слои вроде `app`, `pages`, `widgets`, `features`, `entities`, `shared`. Нижние сегменты не импортируют верхние (направление импортов соблюдать единообразно).
- **Качество кода:** **ESLint 9** + **typescript-eslint** + **eslint-plugin-react** (+ правила hooks по месту) + **Prettier** и **`eslint-config-prettier`**, чтобы не дублировать стилевые правила.

## Контракты, моки и API

1. HTTP-контракты ведите в **TypeSpec** в каталоге контрактов.
2. **`tsp compile`** (или скрипт проекта) выдаёт **OpenAPI** (артефакт YAML/JSON).
3. **Prism** поднимает мок API по этому OpenAPI для локальной разработки фронта и бэка независимо друг от друга (ожидаются цели уровня `make`, например `contracts/generate`, `mock` — после появления скеффолдинга).

Стремитесь, чтобы реализация API соответствовала той же ревизии OpenAPI, что и контракт.

## Тестирование

- **Go:** table-driven тесты, `go test` из `backend/` или через `Makefile`.
- **E2E в браузере:** **Playwright** — либо внутри `frontend/`, либо в **`tests/e2e/`** в корне; при скеффолдинге выберите один вариант и используйте его в документации и CI.

**Перед коммитом:** полный прогон Playwright часто медленный. Разумное разделение: **`make pre-commit`** — быстрые проверки (форматирование при необходимости, линтеры, `go test`, быстрые фронтовые тесты, если есть); **`make test-e2e`** и CI — Playwright, если команда не хочет долгие локальные хуки.

## Сборка, Docker и Render.com

- **Docker:** multi-stage образы и/или **Docker Compose** для `backend`, `frontend`, при необходимости сервиса **Prism**/моков.
- **Render:** деплой на Render (по желанию Blueprint / `render.yaml`). **Render MCP** в Cursor помогает с сервисами и переменными окружения; **секреты и прод-учётки в репозиторий не коммитить**.

## MCP-серверы (Cursor)

Опциональные инструменты для людей и агентов; они **не** заменяют скрипты в репозитории:

- **Render MCP** — деплой и ресурсы Render.
- **Playwright MCP** — сценарии и отладка e2e.
- **Chrome DevTools MCP** — производительность и отладка SPA в браузере.
- **shadcn MCP** — подбор и добавление компонентов shadcn.

Включайте только то, что нужно конкретному разработчику.

## Git-хуки и Makefile

**`Makefile` в корне** считать **единым API команд** проекта (`fmt`, `lint`, `build`, `test`, `pre-commit`, при необходимости `docker-build`, `contracts/*` и т.д.).

- **Husky** настраивает хук **`pre-commit`**, который вызывает **`make pre-commit`**, чтобы логика была в Make, а не дублировалась только в JS.
- Минимальный **`package.json` в корне** может содержать `husky` (и при желании `lint-staged` для ускорения по staged-файлам); тяжёлая работа — через **`make`**.
- Хуки при ошибках должны **`exit non-zero`**, чтобы блокировать коммит при падении линтеров/сборки/тестов.

Ожидаемый смысл целей (имена можно уточнить при скеффолдинге):

- `make fmt` — форматирование Go + Prettier для фронта (запись или только проверка — по договорённости).
- `make lint` — `golangci-lint` + ESLint.
- `make build` — `go build` (или выбранные пакеты) + `vite build`.
- `make test` — тесты Go + быстрые проверки фронта по правилам проекта.
- `make pre-commit` — по порядку: проверка формата (если есть), линт, сборка, тесты (с учётом политики e2e выше).
